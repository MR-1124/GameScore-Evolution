Complete Study Guide — Viva & Exam Preparation

1. Project Overview — What Is This Project?

The Game Score Evolution Simulator is a Python-based web application that models how two competing teams' scores change over time during a match. Instead of using a simple formula like "Team A scores 2 points per minute," this project treats scores as continuously evolving quantities governed by a system of coupled Ordinary Differential Equations (ODEs).

The project has two main academic objectives:

Objective 1: Mathematical Modeling

Build a realistic ODE model that captures attack, defense, momentum, competitive pressure, clutch bursts, and score saturation. The model must produce trajectories that look like plausible game progressions.

Objective 2: Numerical Method Comparison

Implement six different numerical solvers (Euler, Heun, Midpoint, Ralston, Adams-Bashforth 2, RK4) from scratch and compare them on accuracy, runtime, and efficiency using the same ODE model.

"My project models competitive game scoring as a dynamic system using coupled ODEs. I implemented six numerical methods to solve the system and benchmarked them against each other to study the accuracy-versus-speed trade-off in numerical analysis."

2. Why This Project Matters — Academic Significance

This project is academically significant for several reasons:

  • It applies abstract mathematics to a tangible domain. Numerical methods courses often use hypothetical equations. This project applies the same methods to a scenario everyone understands — game scoring — making the concepts concrete and relatable.
  • It demonstrates the entire software engineering pipeline. From mathematical modeling to backend computation to frontend visualization, the project shows how theory turns into a working application.
  • It produces measurable, quantitative results. The benchmark tables provide hard numbers (RMSE, runtime, efficiency score) that can be analyzed and discussed critically.
  • It demonstrates model flexibility. The four scenario presets prove that different game styles can be simulated by adjusting parameters alone, without changing any code or mathematical structure.

Implementing solvers from scratch demonstrates understanding of the algorithms. Libraries hide the mechanics. Hand-coding each method proves you understand the step-by-step process: how slopes are computed, how predictions are corrected, and how errors accumulate. This is the pedagogical value of the project.

3. Understanding ODEs from Scratch

If you need to explain what an ODE is:

What is an ODE?

An Ordinary Differential Equation is an equation that relates a function to its derivative(s). In this project, the function is the score S(t) and the derivative dS/dt is the scoring rate — how fast the score is changing at any given moment.

dS/dt = f(t, S)

"The rate of change of the score at time t
 depends on the current time and the current score."

What is an Initial Value Problem (IVP)?

An IVP is an ODE together with a starting condition. We know the score at t=0 and want to find the score at all future times.

Given: dS/dt = f(t, S)
Given: S(0) = S_initial
Find:  S(t) for t > 0

The key insight: if we know the current state and the derivative, we can estimate the next state by stepping forward a small amount. This is exactly what numerical methods do.

Why can't we solve this analytically?

Our ODE system is nonlinear — the derivative function contains products, max() clamps, and sinusoidal terms. For most nonlinear ODEs, no closed-form solution exists. We must use numerical approximation to step through time and build the solution point by point.

What does "coupled" mean?

A coupled system means the derivative of S_A depends not just on S_A itself, but also on S_B (through the pressure and defensive drag terms). Similarly, dS_B/dt depends on S_A. This means you cannot solve one equation independently — they must be solved simultaneously at each step.

"An ODE describes how a quantity changes over time. My project solves an initial value problem: given starting scores and a derivative function that models scoring dynamics, I step forward in time to compute the full score trajectory. The system is coupled because each team's scoring rate depends on the other team's state."

4. Complete ODE Model — Every Term Explained

The full ODE system has four components. Understanding each one is critical for your viva:

Component 1: Saturation Drive

drive_i = attack_i × momentum_i × tempo × max(0.05, 1 − S_i / score_cap)

What it does: This is the team's base scoring potential. It combines attack power, current momentum, and game tempo. The saturation factor (1 − S_i / score_cap) makes scoring progressively harder as the score approaches the cap — like a governor that prevents unrealistic runaway scoring. The max(0.05, ...) ensures scoring never completely stops.

Example: If attack=1.4, momentum=1.0, tempo=1.0, score=60, cap=120: drive = 1.4 × 1.0 × 1.0 × max(0.05, 1 − 60/120) = 1.4 × 0.5 = 0.7. At half the cap, scoring potential has halved.

Component 2: Competitive Pressure

L = S_A − S_B   (lead difference)
P_A = 1 − pressure_strength × L
P_B = 1 + pressure_strength × L

What it does: When Team A is leading (L > 0), their pressure factor decreases (they ease off) while Team B's increases (they push harder). This models the psychological dynamic of competitive games — the trailing team intensifies play.

Example: If Team A leads by 10 points with pressure_strength=0.03: P_A = 1 − 0.03 × 10 = 0.7, P_B = 1 + 0.03 × 10 = 1.3. The trailing team gets a 30% boost; the leading team is suppressed by 30%.

Component 3: Clutch Oscillation

C_A(t) = clutch_A × sin(0.9 × t)
C_B(t) = clutch_B × sin(0.9 × t + 1.1)

What it does: Creates periodic scoring bursts — moments of hot streaks and cold streaks. The phase offset (1.1 radians ≈ 63°) ensures the two teams don't surge simultaneously. When one team is on a run, the other may be in a scoring lull.

Why sinusoidal? Sine functions are smooth, periodic, bounded (between −1 and +1), and well-understood mathematically. They model cyclical behavior without introducing sharp discontinuities that would make the ODE harder to solve numerically.

Component 4: Defensive Drag

R_A = defense_B × coupling × max(S_A, 0)
R_B = defense_A × coupling × max(S_B, 0)

What it does: A score-proportional resistance force. As a team scores more, the opposing defense exerts more pressure. This is the equilibrium mechanism — without it, scores would grow indefinitely.

Example: If defense_B=1.0, coupling=0.02, S_A=50: drag = 1.0 × 0.02 × 50 = 1.0. This 1.0 is subtracted from Team A's scoring rate, significantly slowing them down.

Final Assembly

dS_A/dt = max(0, drive_A × P_A + C_A(t) − R_A)
dS_B/dt = max(0, drive_B × P_B + C_B(t) − R_B)

The max(0, ...) clamp is the final safeguard: if the combined effect of pressure, clutch, and drag would produce a negative scoring rate, we clamp it to zero. This ensures scores can only increase, which is the physical constraint of a game scoring system.

5. Numerical Methods — Deep Understanding

Each method answers the same question differently: "Given the current state y_n at time t_n, what is the state y_{n+1} at time t_{n+1} = t_n + h?"

Euler Method (1st order)

y_{n+1} = y_n + h × f(t_n, y_n)

Intuition: Take the slope at the current point, draw a straight line forward for distance h, and that's your next point. It's the simplest possible approach — like driving forward while only looking at your current direction, never adjusting for the road ahead.

Weakness: Uses only one slope evaluation at the beginning of the interval. If the true solution curves, Euler misses the curvature and overshoots or undershoots. Error = O(h) per integration — the worst among all methods in this project.

Heun Method / Improved Euler (2nd order)

k1 = f(t_n, y_n)                        ← slope at start
y_predict = y_n + h × k1                ← Euler prediction
k2 = f(t_n + h, y_predict)              ← slope at predicted end
y_{n+1} = y_n + (h/2) × (k1 + k2)      ← average of both slopes

Intuition: Like Euler, but you also look at where Euler would take you and check the slope there. Then you average the two slopes for a better estimate. It's a predictor-corrector: predict with Euler, correct by averaging.

Improvement over Euler: By using two slope evaluations instead of one, the method captures the curvature of the solution and achieves 2nd-order accuracy: O(h²).

Midpoint Method (2nd order)

k1 = f(t_n, y_n)                        ← slope at start
y_mid = y_n + (h/2) × k1                ← half-step prediction
k2 = f(t_n + h/2, y_mid)                ← slope at midpoint
y_{n+1} = y_n + h × k2                  ← use midpoint slope for full step

Intuition: Instead of averaging two endpoints like Heun, go halfway, check the slope there, and use that middle slope for the full step. Often more accurate in practice because the midpoint slope is usually closest to the average slope over the interval.

Ralston Method (optimized 2nd order)

k1 = f(t_n, y_n)
k2 = f(t_n + 3h/4, y_n + (3h/4) × k1)
y_{n+1} = y_n + h × ((1/3) × k1 + (2/3) × k2)

Intuition: Like Midpoint, but uses 3/4 of the interval instead of 1/2, with optimized weights (1/3 and 2/3). These specific constants minimize the leading term of the truncation error. It's still 2nd order, but with a smaller error constant.

Key viva point: Shows that within the same order class, you can optimize the coefficients. Not all 2nd-order methods are equal.

Adams-Bashforth 2 (explicit multistep, 2nd order)

y_{n+1} = y_n + h × ((3/2) × f_n − (1/2) × f_{n-1})

Intuition: Instead of evaluating the derivative multiple times at the current step, reuse the derivative from the previous step. By extrapolating from two consecutive derivative values, you get 2nd-order accuracy with only one new function evaluation per step.

Important detail: AB2 needs two starting values, so it can't self-start. Our implementation uses Heun for the first step (bootstrapping), then switches to AB2 for all subsequent steps.

Trade-off: Fewer function evaluations per step = faster. But it needs memory (previous derivative) and can't self-start. Also, multistep methods can have stability limitations.

Runge-Kutta 4 (4th order)

k1 = f(t_n, y_n)
k2 = f(t_n + h/2, y_n + (h/2) × k1)
k3 = f(t_n + h/2, y_n + (h/2) × k2)
k4 = f(t_n + h, y_n + h × k3)

y_{n+1} = y_n + (h/6) × (k1 + 2×k2 + 2×k3 + k4)

Intuition: Takes FOUR slope evaluations: beginning, two at the midpoint, and one at the end. Combines them with the classic 1:2:2:1 weighting. This captures curvature information so well that the error is O(h⁴) — dramatically smaller than lower-order methods.

Why is it the reference? With 4th-order accuracy, RK4 is overwhelmingly more precise than Euler or Heun for the same step size. Using it with a finer step size (h/refinement) gives an even more accurate result, which we treat as ground truth for benchmarking.

Cost: 4 function evaluations per step vs 1 for Euler. For simple problems, this overhead may not be worth it. The benchmark data shows whether it is.

6. How the Code Works — Architecture Walkthrough

The project has four Python modules and two frontend files. Here's what each does:

model.py — ODE Definitions

Defines the TeamDynamics dataclass (attack, defense, momentum, clutch), the MatchParams dataclass (global parameters), and the score_derivatives() function that computes dS_A/dt and dS_B/dt. This is the mathematical heart of the project.

methods.py — Numerical Solvers

Contains the NumericalMethods class with six static methods: euler, heun, midpoint, ralston, rk4, and adams_bashforth2. Each method calls an internal step function repeatedly through the _integrate() helper. Returns a SimulationResult with time array, score_a array, and score_b array.

analytics.py — Metrics Computation

Implements summarize() to compute game analytics (winner, lead changes, volatility, comeback index) and trajectory_rmse() to compute path accuracy against the reference solution. Defines MatchAnalytics and BenchmarkMetrics dataclasses for structured output.

app.py — Flask Backend & Orchestration

The main application file. Defines scenario presets, method catalog, and the simulate() function that: (1) parses user inputs with type-safe helpers, (2) constructs model parameters, (3) runs the reference RK4 solution, (4) runs each selected method with timing, (5) computes benchmarks, and (6) passes everything to Jinja2 templates for rendering.

app.js — Frontend Interactivity

Handles scenario card selection (updates form values when preset is clicked), method card toggling (checkbox + visual states), and Chart.js initialization for the score evolution line chart and benchmark mixed chart (bar + line).

Templates — HTML Rendering

Four Jinja2 templates: index.html (main simulator), presentation.html (slide deck), study_guide.html (this page), and technical_reference.html (API and formula documentation). All share the same CSS design system with glassmorphic styling.

7. Worked Numerical Example

Let's manually trace one Euler step and one RK4 step using the Balanced Classic scenario defaults:

📝
Setup

t₀ = 0, S_A = 0, S_B = 0, h = 0.1, attack_A = 1.4, defense_A = 1.0, momentum_A = 1.0, clutch_A = 0.2, tempo = 1.0, pressure = 0.03, coupling = 0.02, cap = 120

Computing f(t=0, [0, 0]) for Team A:

# Step 1: Lead and pressure L = 0 − 0 = 0 P_A = 1 − 0.03 × 0 = 1.0 # Step 2: Saturation drive saturation = max(0.05, 1 − 0/120) = max(0.05, 1.0) = 1.0 drive_A = 1.4 × 1.0 × 1.0 × 1.0 = 1.4 # Step 3: Clutch oscillation C_A = 0.2 × sin(0.9 × 0) = 0.2 × 0 = 0.0 # Step 4: Defensive drag R_A = 1.0 × 0.02 × max(0, 0) = 0.0 # Step 5: Final rate dS_A/dt = max(0, 1.4 × 1.0 + 0.0 − 0.0) = 1.4

Euler update:

S_A(0.1) = 0 + 0.1 × 1.4 = 0.14

RK4 would compute at four points:

k₁ at (0, [0,0]), k₂ at (0.05, [0.07, ...]), k₃ at (0.05, [~0.07, ...]), k₄ at (0.1, [~0.14, ...]). The weighted average gives S_A(0.1) ≈ 0.1398 — very close to Euler here because the function is nearly linear near t=0. The difference grows at later time steps when nonlinearity kicks in.

"At t=0 both methods give nearly the same result because the ODE is approximately linear near zero. But as time progresses and scores grow, the saturation, pressure, and clutch terms create nonlinear behavior. That's when Euler starts to diverge from RK4, and the trajectory RMSE shows the accumulated difference."

8. Benchmark Terms You Must Explain Clearly

Trajectory RMSE (Root Mean Square Error) It measures how closely the entire score path matches the reference path, not only the final score. Think of it as measuring the area of deviation between two curves. A method might hit the same final score but follow a completely different path — RMSE catches that path difference. Computed as: √( (1/2N) × Σ [(A_i − A_i_ref)² + (B_i − B_i_ref)²] )
Runtime (milliseconds) The wall-clock time the solver takes to compute all steps. Measured using Python's perf_counter() which provides high-resolution timing. This matters for real-time applications where the solver must complete within a frame budget.
Final-Score Absolute Error The absolute difference between the method's final score and the reference's final score. Simple to understand but insufficient alone — a method could arrive at the right answer via a completely wrong trajectory.
Efficiency Score Defined as 1 / ((RMSE + ε) × (runtime_ms + ε)). Combines accuracy and speed into one number. Higher efficiency means better bang-per-millisecond. The epsilon (ε) prevents division by zero.
Reference Baseline A very fine-step RK4 run is used as the trusted comparison trajectory. The reference step size is h/{{ inputs.benchmark_refinement }}, meaning the reference takes {{ inputs.benchmark_refinement }}× more steps. Since RK4 is 4th-order, this fine reference is extremely close to the true analytic solution.
Lead Changes The number of times the score lead flips from one team to the other during the simulation. More lead changes indicate a more competitive, dynamic match.
Volatility Index Computed as: standard deviation of total score / mean of total score. Higher volatility means the total scoring rate fluctuated more, indicating an unpredictable match with momentum swings.
Comeback Index Measures how big a deficit the eventual winner overcame. Computed as: (max deficit suffered by winner) / (winner's final score). A value near 1.0 means a dramatic, come-from-behind victory.

9. Reading and Interpreting Charts

📊 Score Evolution Chart

  • Each numerical method produces its own pair of lines (Team A = solid, Team B = dashed)
  • Colors are assigned by method name — same color = same solver
  • Overlapping lines = methods agree (the model is well-behaved for those conditions)
  • Diverging lines = methods disagree (reveals where lower-order methods struggle with nonlinearity)
  • Look for divergence points: they often coincide with clutch peaks or sharp pressure transitions

📉 Benchmark Dual-Axis Chart

  • Left Y-axis: Green bars = runtime in milliseconds (taller = slower)
  • Right Y-axis: Orange line = trajectory RMSE (higher = less accurate)
  • The ideal method appears as a short green bar with a low orange point
  • If a method has a tall bar but low RMSE, it's accurate but slow (like RK4)
  • If a method has a short bar but high RMSE, it's fast but inaccurate (like Euler)
⚠️
Important observation for your viva

In close matches, different methods may predict different winners! If Euler says Team A wins but RK4 says Team B wins, it means solver choice literally changed the predicted outcome. This is the strongest possible argument for why numerical method selection matters.

10. Method Comparison — What to Say for Each

Method What to Say About It When It's Useful
Euler "It is the simplest numerical method — one slope evaluation per step. It's easy to implement and explain, but it accumulates error fastest because it cannot capture the curvature of the solution. It serves as the baseline for comparison." Teaching/demonstration; very simple ODEs; when h is extremely small.
Heun "Heun improves on Euler by taking a second slope at the predicted endpoint, then averaging both. This predictor-corrector approach gives 2nd-order accuracy with only one extra function evaluation per step." When you need better accuracy than Euler without much extra cost.
Midpoint "Instead of averaging start and end slopes like Heun, Midpoint estimates the slope at the center of the interval. This often gives better accuracy because the midpoint slope is usually the most representative." General-purpose 2nd-order integration; geometrically intuitive.
Ralston "It's a 2nd-order method with mathematically optimized weights that minimize the leading error term. It shows that even within the same order class, coefficient choice matters." When you want the best possible 2nd-order accuracy.
Adams-Bashforth 2 "AB2 reuses the derivative from the previous step, achieving 2nd-order accuracy with only one new function evaluation. It needs a separate method for bootstrapping the first step." Long simulations where per-step cost matters; when memory is available.
RK4 "RK4 is the gold standard for this project. Four slope evaluations per step give 4th-order accuracy — dramatically better than all other methods here. A finer-step RK4 run serves as the reference trajectory." When accuracy is the priority; reference solutions; scientific computing.

13. How to Answer Common Viva Questions

Q: Why did you choose ODEs instead of a simpler approach? "Because ODEs model continuous change over time. A static formula like 'Team A scores X per minute' cannot capture feedback loops, momentum swings, or competitive pressure. ODEs let me model how the scoring rate itself changes dynamically."
Q: What is the difference between Euler and RK4? "Euler uses one slope evaluation per step and is 1st order accurate. RK4 uses four slope evaluations and is 4th order accurate. In practical terms, if you halve the step size, Euler halves its error while RK4 reduces error by a factor of 16. This means RK4 is dramatically more accurate for the same step size."
Q: Why compare so many methods? "To demonstrate the accuracy-versus-cost trade-off quantitatively. Each method sits at a different point on this spectrum. Comparing all six lets me show that method selection depends on whether you prioritize speed (Euler), accuracy (RK4), or balanced efficiency (possibly Heun or Ralston)."
Q: Why not only use RK4 if it's the most accurate? "Because RK4 costs 4× more function evaluations per step than Euler. For applications with strict runtime budgets — like real-time game physics at 60 FPS — a simpler method with a smaller step size might be more practical. The whole point of numerical analysis is finding the best method for a given context."
Q: What is RMSE and why do you use it? "RMSE stands for Root Mean Square Error. I use trajectory RMSE, which measures how closely an entire score path matches the reference over all time steps. Unlike final-score error, RMSE catches path deviations — a method might reach the correct final score via a completely wrong trajectory."
Q: What makes this a game analytics project, not just a math project? "It computes game-specific metrics: lead changes, volatility index, comeback index, scoring rates. These are meaningful game analytics measures that could be used for match quality evaluation, balance testing, or broadcast commentary."
Q: How is the reference solution generated? "I run RK4 with a step size {{ inputs.benchmark_refinement }}× smaller than the simulation step size. Since RK4 is already 4th-order accurate, using a much finer step makes the reference solution extremely close to the true analytic solution — close enough to serve as ground truth for benchmarking."
Q: Can this model be used for real game prediction? "The model structure is suitable, but the parameters would need to be calibrated against real game data using statistical fitting or machine learning. Currently, the parameters are chosen to produce realistic-looking trajectories, not to match any specific real-world dataset."
Q: What would you improve if you had more time? "Three things: (1) Add stochastic noise terms to model game unpredictability, (2) Implement adaptive step-size control like RK45 that adjusts h based on estimated local error, and (3) Add time-varying parameters to model fatigue and tactical changes during a match."
Q: Explain the coupling in the ODE system. "Both equations reference the other team's state. Team A's pressure factor depends on Team B's score through the lead difference L = S_A − S_B. Team A's defensive drag uses Team B's defense coefficient. This means you cannot solve one equation independently — they must be integrated simultaneously, which is why numerical methods process both components at each step."

15. Glossary of Technical Terms

ODE (Ordinary Differential Equation)
An equation relating a function to its derivatives. In this project, the function is the score and the derivative is the scoring rate.
Initial Value Problem (IVP)
An ODE together with a known starting condition. We know S(0) and want to find S(t) for all t > 0.
Coupled System
A system where each equation depends on the solution of another equation. Here, dS_A/dt depends on S_B and vice versa.
Step Size (h)
The time increment between consecutive solver evaluations. Smaller h = more accurate but more computational work.
Truncation Error
The error introduced by approximating the continuous differential equation with discrete steps. It comes from truncating the Taylor series.
Order of Accuracy
Describes how fast error decreases as h shrinks. An order-p method has global error O(h^p). Halving h reduces error by factor 2^p.
RMSE (Root Mean Square Error)
A measure of the average deviation between two trajectories over time. Computed as the square root of the mean squared differences.
Predictor-Corrector
A two-phase method: predict using a simple formula (like Euler), then correct using additional information (like the slope at the predicted point).
Multistep Method
A method that uses information from multiple previous steps (not just the current one). Adams-Bashforth 2 uses derivatives from steps n and n−1.
Bootstrapping
Using a different method (like Heun) to compute the first step required by a multistep method that can't self-start.
Saturation
The tendency for growth to slow as a quantity approaches a limit. In this project, scoring slows as it nears the score cap.
Efficiency Score
A composite metric: 1 / (RMSE × runtime). Higher values indicate better accuracy per unit of computation time.
