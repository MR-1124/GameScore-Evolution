Key Implementation Details

Type Safety

The _float() and _int() helper functions wrap all form parsing with try/except fallback to default values. This prevents crashes from malformed inputs.

def _float(form, key, default): try: return float(form.get(key, default)) except (TypeError, ValueError): return default

Reference Sampling

The reference RK4 runs at finer resolution (h/refinement, steps × refinement). To compare against coarse solutions, the reference is downsampled by picking every refinement-th point:

def _sample_reference(series, refinement, count): sampled = [] for i in range(count): idx = min(i * refinement, len(series) - 1) sampled.append(series[idx]) return sampled

2. Core State Variable and Model Definition

The simulator tracks a 2D state vector over time, where each component represents one team's score:

State Vector 
y(t) = [S_A(t), S_B(t)]

S_A(t) = Team A score at time t    (non-negative real number)
S_B(t) = Team B score at time t    (non-negative real number)

Initial condition: y(0) = [S_A(0), S_B(0)]    (user-specified or preset)

The system is a 2-dimensional, first-order, autonomous-coupled, nonlinear ODE system with monotone constraints. Every numerical solver processes both components simultaneously at each step, maintaining the coupling relationship.

3. Full ODE Structure — Complete Derivation

Step 1 — Lead Difference 
L = S_A − S_B

Interpretation: Positive L means Team A leads.
               Negative L means Team B leads.
               Zero means tied.
Step 2 — Pressure Response 
P_A = 1 − pressure_strength × L
P_B = 1 + pressure_strength × L

When Team A leads (L > 0):
  P_A < 1 (Team A slows down — complacency effect)
  P_B > 1 (Team B speeds up — chase effect)

When tied (L = 0):
  P_A = P_B = 1 (no pressure bias)
Step 3 — Saturation Drive (per team) 
saturation_i = max(0.05, 1 − S_i / score_cap)
drive_i = attack_i × momentum_i × tempo × saturation_i

The saturation factor creates a logarithmic-like growth curve:
  At S_i = 0:        saturation = 1.0    (full power)
  At S_i = cap/2:    saturation = 0.5    (half power)
  At S_i = cap:      saturation = 0.05   (minimum power, not zero)

This prevents unrealistic linear growth and models score-dependent difficulty.
Step 4 — Clutch Oscillation 
C_A(t) = clutch_A × sin(0.9 × t)
C_B(t) = clutch_B × sin(0.9 × t + 1.1)

Properties:
  Frequency:    0.9 rad/time_unit → period ≈ 2π/0.9 ≈ 6.98 time units
  Amplitude:    bounded by clutch coefficient
  Phase offset: 1.1 rad ≈ 63° between teams
  Effect:       adds positive/negative bursts to scoring rate
  Combined with max(0, ...): negative clutch can suppress scoring but not reverse it
Step 5 — Defensive Drag (cross-team) 
R_A = defense_B × coupling × max(S_A, 0)
R_B = defense_A × coupling × max(S_B, 0)

Key properties:
  - Proportional to current score (higher scores face more resistance)
  - Uses OPPONENT's defense rating (Team B defends against Team A)
  - Coupling parameter scales the interaction strength
  - max(S, 0) prevents negative score from creating negative drag
Step 6 — Final ODE Assembly 
dS_A/dt = max(0, drive_A × P_A + C_A(t) − R_A)
dS_B/dt = max(0, drive_B × P_B + C_B(t) − R_B)

The max(0, ...) clamp is applied to the ENTIRE expression, not individual terms.
This means negative clutch or high drag can reduce the rate to zero but never below.
Scores are monotonically non-decreasing by construction.

5. Numerical Methods — Formulas and Implementation

For all methods below: f(t, y) is the ODE derivative function returning (dS_A/dt, dS_B/dt), and states are clipped to non-negative after each step via _clip_state(y) = (max(0, y[0]), max(0, y[1])).

5.1 Euler (1st order, 1 evaluation/step)

Update Formula 
y_{n+1} = y_n + h × f(t_n, y_n)
# Implementation (methods.py) def _euler_step(f, t, y, h): dy = f(t, y) return y[0] + h * dy[0], y[1] + h * dy[1]

Local truncation error: O(h²). Global error: O(h). The method is derived from the first two terms of the Taylor expansion: y(t+h) = y(t) + h·y'(t) + O(h²).

5.2 Heun / Improved Euler (2nd order, 2 evaluations/step)

Update Formula 
k1 = f(t_n, y_n)
y_predict = y_n + h × k1
k2 = f(t_n + h, y_predict)
y_{n+1} = y_n + (h/2) × (k1 + k2)
# Implementation (methods.py) def _heun_step(f, t, y, h): k1 = f(t, y) predictor = (y[0] + h * k1[0], y[1] + h * k1[1]) k2 = f(t + h, predictor) return ( y[0] + (h / 2.0) * (k1[0] + k2[0]), y[1] + (h / 2.0) * (k1[1] + k2[1]), )

Derivation: Heun is equivalent to the trapezoidal rule for integration: ∫f ≈ (h/2)(f(a) + f(b)). The k1 is f at the start, k2 is f at the Euler-predicted end.

5.3 Midpoint RK2 (2nd order, 2 evaluations/step)

Update Formula 
k1 = f(t_n, y_n)
y_mid = y_n + (h/2) × k1
k2 = f(t_n + h/2, y_mid)
y_{n+1} = y_n + h × k2
# Implementation (methods.py) def _midpoint_step(f, t, y, h): k1 = f(t, y) mid = (y[0] + 0.5 * h * k1[0], y[1] + 0.5 * h * k1[1]) k2 = f(t + h / 2.0, mid) return y[0] + h * k2[0], y[1] + h * k2[1]

Derivation: Equivalent to the midpoint rule for integration: ∫f ≈ h × f((a+b)/2). Uses the slope at the center of the interval for the full step.

5.4 Ralston RK2 (optimized 2nd order, 2 evaluations/step)

Update Formula 
k1 = f(t_n, y_n)
k2 = f(t_n + 3h/4, y_n + (3h/4) × k1)
y_{n+1} = y_n + h × ((1/3) × k1 + (2/3) × k2)
# Implementation (methods.py) def _ralston_step(f, t, y, h): k1 = f(t, y) y2 = (y[0] + 0.75 * h * k1[0], y[1] + 0.75 * h * k1[1]) k2 = f(t + 0.75 * h, y2) return ( y[0] + h * ((1.0/3.0) * k1[0] + (2.0/3.0) * k2[0]), y[1] + h * ((1.0/3.0) * k1[1] + (2.0/3.0) * k2[1]), )

Optimization: The 1/3 and 2/3 weights, and the 3/4 evaluation point, are chosen to minimize the leading coefficient of the O(h³) truncation error term. The result is the same order as Heun and Midpoint but with a smaller error constant.

5.5 Adams-Bashforth 2 (explicit multistep, 2nd order)

Update Formula 
y_{n+1} = y_n + h × ((3/2) × f(t_n, y_n) − (1/2) × f(t_{n-1}, y_{n-1}))

Bootstrapping: Step 0→1 uses Heun method
Subsequent:    Steps 1→2, 2→3, ... use AB2 formula
# Implementation (methods.py) — simplified logic # Bootstrap with Heun for first step y1 = _clip_state(_heun_step(f, t0, y0, h)) # AB2 loop for remaining steps for _ in range(1, steps): fn = f(t, curr) fnm1 = f(t - h, prev) next_y = ( curr[0] + h * (1.5 * fn[0] - 0.5 * fnm1[0]), curr[1] + h * (1.5 * fn[1] - 0.5 * fnm1[1]), ) prev, curr = curr, _clip_state(next_y)

Derivation: AB2 approximates the integral ∫f dt by fitting a polynomial through the two most recent derivative values and extrapolating. The coefficients 3/2 and −1/2 come from the Lagrange interpolating polynomial.

Key advantage: Only 1 new function evaluation per step (it reuses the stored previous derivative). Key disadvantage: requires a startup step and can have stability limitations compared to single-step methods.

5.6 Runge-Kutta 4 (4th order, 4 evaluations/step)

Update Formula 
k1 = f(t_n, y_n)
k2 = f(t_n + h/2, y_n + (h/2) × k1)
k3 = f(t_n + h/2, y_n + (h/2) × k2)
k4 = f(t_n + h, y_n + h × k3)

y_{n+1} = y_n + (h/6) × (k1 + 2k2 + 2k3 + k4)
# Implementation (methods.py) def _rk4_step(f, t, y, h): k1 = f(t, y) k2 = f(t + h/2.0, (y[0] + h*k1[0]/2.0, y[1] + h*k1[1]/2.0)) k3 = f(t + h/2.0, (y[0] + h*k2[0]/2.0, y[1] + h*k2[1]/2.0)) k4 = f(t + h, (y[0] + h*k3[0], y[1] + h*k3[1])) return ( y[0] + (h/6.0) * (k1[0] + 2.0*k2[0] + 2.0*k3[0] + k4[0]), y[1] + (h/6.0) * (k1[1] + 2.0*k2[1] + 2.0*k3[1] + k4[1]), )

Derivation: RK4 is derived from matching the Taylor expansion through the h⁴ term. The 1:2:2:1 weights and the specific evaluation points are chosen so that the method agrees with the Taylor series through O(h⁵), giving O(h⁴) global error.

Why it's the reference: With 4th-order accuracy, RK4 is the most precise method in this collection. Using it at h/refinement step size gives an even more accurate trajectory that serves as ground truth for benchmarking.

6. Convergence Analysis — Order of Accuracy Theory

The order of accuracy of a numerical method describes how quickly its error decreases when the step size h is reduced:

Error Scaling Laws 
Local Truncation Error (per step):
  Euler:      LTE = O(h²)     — error per step scales as h²
  Heun/RK2:   LTE = O(h³)     — scales as h³
  RK4:        LTE = O(h⁵)     — scales as h⁵

Global Error (accumulated over N = T/h steps):
  Euler:      GE = O(h)       — first-order convergence
  Heun/RK2:   GE = O(h²)      — second-order convergence
  RK4:        GE = O(h⁴)      — fourth-order convergence

Practical implication — halving h:
  Euler:      error halves           (factor 2)
  Heun/RK2:   error quarters         (factor 4)
  RK4:        error reduces by 16×   (factor 16)
📐
Why global error is one order less than local

Local error accumulates over N = T/h steps. If LTE = O(h^{p+1}) per step, then after N = T/h steps: Global Error = N × LTE = (T/h) × O(h^{p+1}) = O(h^p). This is why Euler has O(h²) local but O(h) global error.

Practical Convergence in This Project

You can verify convergence by running the simulator with different step sizes and observing how RMSE changes:

Method h = 0.2 h = 0.1 (half) Expected Ratio
Euler (O(h))RMSE₁≈ RMSE₁ / 22:1
Heun (O(h²))RMSE₂≈ RMSE₂ / 44:1
RK4 (O(h⁴))RMSE₃≈ RMSE₃ / 1616:1

If your benchmark data shows approximately these ratios, it confirms the theoretical convergence orders of the implemented methods.

7. Stability Concepts

Stability determines whether errors grow or decay as the method iterates. An unstable method will produce results that diverge wildly even if the true solution is bounded.

What is a Stability Region?

For a test equation dy/dt = λy, a method is stable when h × λ falls within its stability region in the complex plane. Outside this region, numerical errors amplify exponentially.

For this project's ODE, the effective "λ" varies over time (due to nonlinearity), but the key insight is: if you make h too large, any method can become unstable.

Why This Matters Here

Our ODE has bounded derivatives (due to saturation and max(0,...)), so stability is generally not a critical issue with reasonable step sizes (h ≤ 0.2). However, if a user sets very large h with high attack coefficients, Euler might overshoot while RK4 remains stable due to its larger stability region.

This is another advantage of higher-order methods: they typically have larger stability regions, allowing bigger step sizes without instability.

💡
Viva tip

If asked about stiff ODEs: this model is NOT stiff because all terms are bounded and the effective eigenvalues of the Jacobian are moderate. Stiff ODEs require implicit methods (like backward Euler or BDF) which this project does not implement — that would be a valid future enhancement.

8. Adams-Bashforth 2 — Detailed Analysis

AB2 deserves special attention because it's fundamentally different from the single-step Runge-Kutta family:

How Multistep Methods Work

Instead of evaluating f multiple times at the current step (like RK methods), multistep methods evaluate f once but combine it with stored values from previous steps. AB2 uses:

y_{n+1} = y_n + h × ((3/2)f_n − (1/2)f_{n-1})

This is a linear combination of:
  f_n     = derivative at current step    (weight: +3/2)
  f_{n-1} = derivative at previous step   (weight: -1/2)

The coefficients 3/2 and −1/2 come from fitting a linear interpolating polynomial through (t_{n-1}, f_{n-1}) and (t_n, f_n), then extrapolating forward to t_{n+1}.

Bootstrap Requirement

AB2 needs two starting points (y₀ and y₁) but is only given y₀. The implementation bootstraps by using Heun's method for the first step, providing y₁ with 2nd-order accuracy. All subsequent steps use the AB2 formula.

Why Heun for bootstrap? Because it matches AB2's 2nd-order accuracy. Using Euler (1st order) for the bootstrap step would introduce a 1st-order error that contaminates the entire solution.

Advantages over RK2 Methods

  • Cost: Only 1 new function evaluation per step (Heun/Midpoint need 2)
  • Efficiency: For long simulations, the reduced evaluation count saves significant time
  • Memory: Only needs to store one previous derivative value

Disadvantages

  • Cannot self-start: Requires a separate method for the first step
  • Smaller stability region: More susceptible to instability than some RK methods
  • Startup error: Bootstrap step quality affects all subsequent steps
  • Not self-correcting: Unlike predictor-corrector methods, AB2 doesn't check its own accuracy

9. Benchmark Metrics — Complete Specification

Each method is evaluated with five quantitative metrics:

9.1 Final-Score Absolute Error

Error_A = |S_A^{method}(T) − S_A^{ref}(T)|
Error_B = |S_B^{method}(T) − S_B^{ref}(T)|

T = total simulation time = h × steps

Simple to compute and understand, but insufficient alone — a method could accumulate significant mid-trajectory error that cancels by the end.

9.2 Trajectory RMSE

RMSE = √( (1/(2N)) × Σ_{i=0}^{N} [ (A_i − A_i^{ref})² + (B_i − B_i^{ref})² ] )

where N = total number of comparison points (steps + 1)
factor of 2N: divides by 2 because there are 2 state variables

This is the primary accuracy metric. It measures the average path deviation over the entire simulation, capturing mid-trajectory drift that final-score error misses.

9.3 Runtime

start = perf_counter()
result = solver(derivative, t0, initial_state, h, steps)
runtime_ms = (perf_counter() − start) × 1000.0

Wall-clock time in milliseconds. Uses Python's perf_counter() for maximum precision. Note: runtime can vary between runs due to OS scheduling; the values are indicative, not exact.

9.4 Efficiency Score

Efficiency = 1 / ((RMSE + ε) × (runtime_ms + ε))

where ε = 1×10⁻⁶ (prevents division by zero)

Higher = better. A method that is both fast and accurate will have a high efficiency score. This single metric captures the speed-accuracy trade-off that is central to numerical method selection.

9.5 Best-of Rankings

After computing all metrics, the system identifies:

  • Best accuracy: Method with lowest trajectory_rmse
  • Fastest: Method with lowest runtime_ms
  • Best efficiency: Method with highest efficiency_score

Final Scores S_A[-1], S_B[-1] End-of-match score for each team Whether methods agree on the outcome

14. Comprehensive Viva Reference

A structured approach to defending this project in an academic evaluation:

Start with the "Why"

Explain why ODEs are needed: game scoring is continuous, interactive, and nonlinear. Simple formulas can't capture feedback loops. Numerical methods let us compute trajectories when no closed-form solution exists.

Walk Through the Model

Explain each ODE term: drive (offensive engine), pressure (competitive adjustment), clutch (periodic bursts), drag (defensive damping). Emphasize coupling and the max(0) constraint.

Name and Describe Each Solver

For each of the 6 methods, give a one-sentence description and state its order. Explain that higher order means faster error reduction when h decreases.

Show Benchmark Results

Point to RMSE, runtime, and efficiency. Identify which method wins each category. Discuss the trade-off: fastest isn't most accurate, and most accurate isn't most efficient.

Demonstrate Scenarios

Switch between presets to show model flexibility. Point out how coefficient changes produce different trajectories without structural code changes.

Address Limitations Head-On

Acknowledge: deterministic (no randomness), fixed parameters, single-frequency clutch, no adaptive stepping. Then state potential improvements: SDEs, RKF45, time-varying params.

Conclude with the Core Insight

"Solver choice affects not just accuracy and speed, but potentially the predicted match outcome itself. This project demonstrates that numerical method selection is a consequential engineering decision."
