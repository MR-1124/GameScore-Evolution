Complete Technical Reference

This page documents all input fields, parameter interpretation, solver formulas, benchmarking equations, and implementation details of the Game Score Evolution Simulator.

Use this as your definitive explanation sheet for viva and teacher discussion.

1. Core State and Model Variables

The simulator tracks two state variables over time: Team A score \(S_A(t)\) and Team B score \(S_B(t)\). The state vector is: 

y(t) = [S_A(t), S_B(t)]

The system is coupled: each derivative depends on both scores, not only its own team.

2. Full ODE Structure

Lead difference: L = S_A - S_B

Pressure terms:
P_A = 1 - pressure_strength * L
P_B = 1 + pressure_strength * L

Saturation drive:
Drive_i = attack_i * momentum_i * tempo * max(0.05, 1 - S_i / score_cap)

Clutch oscillation:
C_A(t) = clutch_A * sin(0.9 * t)
C_B(t) = clutch_B * sin(0.9 * t + 1.1)

Defensive drag:
R_A = defense_B * coupling * S_A
R_B = defense_A * coupling * S_B

Final ODE rates:
dS_A/dt = max(0, Drive_A * P_A + C_A(t) - R_A)
dS_B/dt = max(0, Drive_B * P_B + C_B(t) - R_B)

The \(max(0, \cdot)\) clamp ensures the model does not produce negative scoring rates.

4. Numerical Methods and Update Formulas

For each method, y_n is the state at step n and f(t, y) is the ODE derivative.

Euler (1st order)

y_{n+1} = y_n + h * f(t_n, y_n)

Heun / Improved Euler (2nd order)

k1 = f(t_n, y_n)
y_predict = y_n + h * k1
k2 = f(t_n + h, y_predict)
y_{n+1} = y_n + (h/2) * (k1 + k2)

Midpoint RK2 (2nd order)

k1 = f(t_n, y_n)
k2 = f(t_n + h/2, y_n + (h/2) * k1)
y_{n+1} = y_n + h * k2

Ralston RK2 (optimized 2nd order)

k1 = f(t_n, y_n)
k2 = f(t_n + 3h/4, y_n + (3h/4) * k1)
y_{n+1} = y_n + h * ((1/3) * k1 + (2/3) * k2)

Adams-Bashforth 2 (explicit multistep)

y_{n+1} = y_n + h * ( (3/2) * f_n - (1/2) * f_{n-1} )

AB2 requires one starting step; this implementation bootstraps with Heun.

RK4 (4th order)

k1 = f(t_n, y_n)
k2 = f(t_n + h/2, y_n + (h/2) * k1)
k3 = f(t_n + h/2, y_n + (h/2) * k2)
k4 = f(t_n + h, y_n + h * k3)

y_{n+1} = y_n + (h/6) * (k1 + 2k2 + 2k3 + k4)

5. Benchmark Metrics and Equations

Each selected method is evaluated against a fine-step RK4 reference trajectory.

Final-score absolute error

Error_A = |S_A^{method}(T) - S_A^{ref}(T)|
Error_B = |S_B^{method}(T) - S_B^{ref}(T)|

Trajectory RMSE

RMSE = sqrt( (1/(2N)) * sum_{i=1..N} [ (A_i - A_i_ref)^2 + (B_i - B_i_ref)^2 ] )

Efficiency score

Efficiency = 1 / ((RMSE + epsilon) * (runtime_ms + epsilon))

Higher efficiency means better speed-accuracy balance.

8. Implementation Details (Code Architecture)

  • Model: ODE definitions and team/match parameter classes in game_score_evolution/model.py
  • Solvers: all numerical methods in game_score_evolution/methods.py
  • Analytics: gameplay metrics and RMSE helper in game_score_evolution/analytics.py
  • Backend: request parsing, scenario defaults, simulation loop, benchmark generation in app.py
  • Frontend: controls, scenario cards, method cards, chart rendering in templates/index.html and static/app.js
  • Presentation pages: teacher-facing and student-facing explainers in templates/presentation.html and templates/study_guide.html

9. Practical Interpretation for Viva

  1. Start with model logic: explain why ODEs are needed.
  2. Explain each input as a coefficient that changes trajectory shape.
  3. Run at least two methods and compare trajectory differences.
  4. Use benchmark chart to justify speed-versus-accuracy tradeoff.
  5. Use scenario presets to prove model flexibility.
  6. Conclude that solver selection depends on fidelity requirement and runtime budget.
